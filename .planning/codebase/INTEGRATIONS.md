# External Integrations

**Analysis Date:** 2026-03-09

## APIs & External Services

**Data Availability Verification:**
- Data Availability Prover (off-chain) - Signs assertion data availability proofs
  - SDK/Client: Custom ECDSA signature (implemented in `DAVerifierECDSA.sol`)
  - Verification method: ECDSA signature recovery using `solady/ECDSA.recoverCalldata`
  - Integration point: `src/verification/da/DAVerifierECDSA.sol`

**External Reference:**
- Assertion DA (https://github.com/phylaxsystems/assertion-da) - Referenced for DA proof generation
  - Not directly imported; users obtain signatures from this external system

**Off-Chain Storage:**
- Assertion bytecode stored in Assertion DA system (external service)

**File Storage:**
- Not applicable - no file storage integrations

**Caching:**
- Not applicable - no caching layer integrations

## Authentication & Identity

**Admin Verification:**
- Multiple admin verifier implementations for adopter identity verification:
  1. `AdminVerifierOwner` (`src/verification/admin/AdminVerifierOwner.sol`)
     - Verifies admin via `owner()` call on adopter contract
     - Silent failure if adopter does not expose compatible `owner()` method
  2. `AdminVerifierWhitelist` (`src/verification/admin/AdminVerifierWhitelist.sol`)
     - Owner-managed whitelist mapping of adopter -> allowed admin
     - Supports exclusion flow with configurable releaser

**Access Control:**
- Role-based access control via OpenZeppelin `AccessControl`
- Role hierarchy defined in `StateOracleAccessControl.sol`:
  - `owner()` / `DEFAULT_ADMIN_ROLE` - Protocol owner (Cold Multisig pattern)
  - `GOVERNANCE_ROLE` - Protocol configuration management
  - `GUARDIAN_ADMIN_ROLE` - Guardian management
  - `GUARDIAN_ROLE` - Emergency defensive actions
  - `OPERATOR_ADMIN_ROLE` - Operator management
  - `OPERATOR_ROLE` - Whitelist operations

**No External Auth Provider:**
- Custom implementation; no third-party auth service integration

## CI/CD & Deployment

**Hosting:**
- Smart contracts deployed to Ethereum or EVM-compatible blockchains
- No specific cloud hosting dependency

**Deployment Scripts:**
- `script/DeployCore.s.sol` - Standard deployment flow
  - Deploys DA verifier, admin verifiers, StateOracle implementation, and proxy
  - Funds three persistent addresses with 1 wei if empty
- `script/DeployCoreWithCreateX.s.sol` - Deterministic CreateX deployment
  - Uses CreateX at fixed address `0xba5Ed099633D3B313e4D5F7bdc1305d3c28ba5Ed`
  - Generates deployer-derived salts for reproducible addresses
- `script/DeployCoreWithStaging.s.sol` - Shared infrastructure for production + staging
  - Deploys shared DA verifier and admin verifiers
  - Deploys two oracle instances (production and staging)

**CI Pipeline:**
- Not configured
- Manual script execution via `forge script`
- Tests run locally via `forge test`

**Build Artifacts:**
- ABI artifacts generated by `shell/create_artifacts.sh`
  - Runs `forge build` to generate contract ABIs
  - Extracts ABIs to `artifacts/` for npm publishing
  - Subdirectories: `artifacts/`, `artifacts/interfaces/`, `artifacts/libraries/`

## Environment Configuration

**Required env vars:**
- `STATE_ORACLE_MAX_ASSERTIONS_PER_AA` - Max assertions per adoption adopter
- `STATE_ORACLE_ASSERTION_TIMELOCK_BLOCKS` - Assertion activation/deactivation delay
- `STATE_ORACLE_ADMIN_ADDRESS` - Initial protocol admin address
- `DA_PROVER_ADDRESS` - Address authorized to sign DA proofs
- `DEPLOY_ADMIN_VERIFIER_OWNER` - Boolean (true/false) for owner verifier deployment
- `DEPLOY_ADMIN_VERIFIER_WHITELIST` - Boolean (true/false) for whitelist verifier deployment
- `ADMIN_VERIFIER_WHITELIST_ADMIN_ADDRESS` - Required if whitelist verifier enabled

**Secrets location:**
- Not stored in repository (respects `.gitignore`)
- Secrets provided via environment variables at deployment/test time
- No `.env` file committed (security practice)

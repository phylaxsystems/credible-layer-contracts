# AGENTS.md

This file gives coding agents repo-specific guidance for working safely and collaboratively in `credible-layer-contracts`.

The repository is small, security-sensitive, and centered on an upgradeable Solidity system. The right default behavior is:

1. Read the relevant contracts and tests first.
2. Make the smallest change that fully solves the task.
3. Preserve upgradeability and access-control invariants.
4. Add or update tests with every behavior change.
5. Avoid touching generated or vendored files unless the task explicitly requires it.

## What This Repo Is

This repository contains the core smart contracts for the Credible Layer protocol. The primary component is an upgradeable `StateOracle` that:

- registers assertion adopters,
- tracks manager-controlled assertion lifecycles,
- verifies protocol admins through pluggable admin verifiers,
- verifies data availability through a DA verifier,
- enforces timelocks and per-adopter assertion limits,
- exposes emergency and operational controls through role-based access control.

The project is built with Foundry. `package.json` exists mostly as a thin wrapper for Foundry commands and npm publishing.

## Primary Tooling

- Language: Solidity `^0.8.28`
- Test/build framework: Foundry
- Libraries: OpenZeppelin, Solady, forge-std

## Architecture Overview

### 1. `StateOracle`

`StateOracle` is the center of the protocol.

Responsibilities:

- register an assertion adopter after admin verification,
- record the adopter's manager,
- add and remove assertions with a timelock,
- enforce `maxAssertionsPerAA`,
- verify data availability through `DA_VERIFIER`,
- maintain a registry of allowed admin verifiers,
- maintain an optional caller whitelist,
- expose governance, guardian, and operator actions.

Important details:

- It is deployed as an implementation behind `TransparentUpgradeableProxy`.
- The constructor sets immutable configuration and then disables initialization on the implementation.
- Real initialization happens through `initialize(...)` on the proxy.
- `whitelistEnabled` is set to `true` during initialization.
- `hasAssertion()` only checks whether the assertion was ever added, not whether it is currently active.
- Removed assertions cannot be re-added because their `activationBlock` remains non-zero.
- `assertionCount` tracks currently registered, not-yet-removed assertions for each adopter.

### 2. `StateOracleAccessControl`

This contract encodes a non-trivial security model. Treat it as an invariant-heavy area.

Role hierarchy:

- `owner()` is tightly coupled to `DEFAULT_ADMIN_ROLE`
- `GOVERNANCE_ROLE` manages protocol configuration
- `GUARDIAN_ADMIN_ROLE` manages guardians
- `GUARDIAN_ROLE` handles emergency defensive actions
- `OPERATOR_ADMIN_ROLE` manages operators
- `OPERATOR_ROLE` handles whitelist operations

Critical invariant:

- `owner()` and `DEFAULT_ADMIN_ROLE` must remain coupled 1:1.

That invariant is intentionally enforced by custom overrides of:

- `_transferOwnership`
- `grantRole`
- `revokeRole`
- `renounceRole`

If you touch this contract, read it fully and update tests accordingly.

### 3. Admin Verifiers

Current verifier implementations:

- `AdminVerifierOwner`: validates adminship by calling `owner()` on the adopter contract
- `AdminVerifierWhitelist`: owner-managed mapping of adopter -> allowed admin, with exclusion flow
- `AdminVerifierSuperAdmin`: test-only verifier that approves a single super admin for every adopter

Important behavior:

- `AdminVerifierOwner` silently returns `false` if the adopter does not expose a compatible `owner()` method.
- `AdminVerifierWhitelist.exclude(...)` removes any current whitelist entry and blocks re-whitelisting until the configured releaser clears the exclusion.
- `AdminVerifierSuperAdmin` is not a production-safe verifier. Keep it test-only unless the user explicitly asks otherwise.

### 4. DA Verifier

`DAVerifierECDSA` approves an assertion when `ECDSA.recoverCalldata(assertionId, proof)` matches the configured `DA_PROVER`.

Important detail:

- `metadata` is currently ignored by `DAVerifierECDSA`; it is passed for interface compatibility.

### 5. Batch Execution

`Batch.batch(bytes[] calldata calls)` executes each encoded call via `delegatecall` into `address(this)`.

Implications:

- state changes occur in the same contract context,
- failures are wrapped as `BatchError(bytes result)`,
- methods invoked through `batch` must not rely on `msg.value`.

If you change batchable functions, consider both direct calls and batched calls.

## Upgradeability Rules

This repo already uses a proxy-based deployment model. Do not make storage-breaking changes casually.

When editing upgradeable contracts:

- never reorder existing storage variables,
- never remove existing storage variables,
- never change the meaning of an existing storage slot without explicit migration planning,
- append new storage variables only at the end of storage layout,
- remember that `immutable` values live in implementation bytecode, not proxy storage,
- keep constructor logic limited to immutable setup and initializer lockout,
- keep initialization logic in `initialize(...)`.

`StateOracle` inherits from multiple contracts. Be careful with inheritance order and added storage in parent contracts.

If a task requires a storage layout change, call it out clearly and add focused tests.

## Testing Expectations

This codebase is security-sensitive. Behavior changes should come with tests.

Use these slices depending on what changed:

- `forge test --match-path test/StateOracle.t.sol`
  For assertion registration, manager flow, whitelist toggles, adopter behavior.
- `forge test --match-path test/StateOracleAccessControl.t.sol`
  For ownership, role hierarchy, role transfer, and invariant changes.
- `forge test --match-path test/AdminVerifierWhitelist.t.sol`
  For whitelist verifier behavior.
- `forge test --match-path test/AdminVerifierSuperAdmin.t.sol`
  For test-only super-admin verifier behavior.
- `forge test --match-path test/DAVerifierECDSA.t.sol`
  For signature verification logic.
- `forge test --match-path test/Batch.t.sol`
  For batch behavior.
- `forge test --match-path test/integration/*`
  For deployment-flow and higher-level integration changes.

Safe default before finishing a non-trivial change:

- `forge fmt`
- `forge test`

If the change affects published ABI output, also run:

- `npm run prepare`

## Deployment And Script Guidance

Deployment scripts live in `script/`.

Main ones:

- `DeployCore.s.sol`: standard deployment flow
- `DeployCoreWithCreateX.s.sol`: deterministic CreateX deployment flow
- `DeployCoreWithStaging.s.sol`: deploys shared verifier infrastructure plus production and staging oracles

Important deployment facts:

- `DeployCore` funds three persistent addresses with `1 wei` when empty.
- `DeployCoreWithCreateX` uses a fixed CreateX address and deployer-derived salts.
- `DeployCoreWithStaging` shares the same DA verifier and admin verifiers across two oracle instances.
- `TransparentUpgradeableProxy` is used, so proxy admin behavior matters in tests and scripts.

Environment variables documented in `README.md` are part of the operational interface. If a script change alters required configuration, update the docs in the same task.

## Generated ABI Artifacts

The release workflow publishes ABIs generated from `shell/create_artifacts.sh`.

If you change:

- any public or external function,
- events,
- custom errors that downstream consumers inspect,
- interface definitions,
- the set of contracts exported in `artifacts/`,

then regenerate `artifacts/` and include the updated outputs only if the task expects generated files to be committed.

Do not manually edit JSON artifacts.

## Repo-Specific Pitfalls

### Transparent proxy admin gotcha

The proxy admin cannot interact with implementation functions through the transparent proxy the same way as normal users. Tests explicitly avoid proxy-admin callers with helper logic like `noAdmin(...)`.

If a test unexpectedly fails around access control, check whether the caller is the proxy admin rather than assuming the contract logic is wrong.

### Whitelist defaults

`StateOracle.initialize(...)` enables the whitelist immediately.

Many tests disable it in `setUp()` to isolate the behavior being tested. If you add tests, decide explicitly whether whitelist behavior should be in scope.

### Assertion lifecycle semantics

An assertion:

- becomes active at `block.number + ASSERTION_TIMELOCK_BLOCKS`,
- becomes inactive at `block.number + ASSERTION_TIMELOCK_BLOCKS` when removed,
- cannot be added twice,
- cannot be re-added after removal.

Do not accidentally "fix" this without confirming product intent; the README and implementation currently agree on this one-time registration rule.

### `AdminVerifierOwner` compatibility

Only contracts exposing a compatible `owner()` view will pass `AdminVerifierOwner`.

If a task involves non-ownable adopters, the likely solution is a new verifier, not loosening the existing one.

### SPDX and style consistency

This repo already mixes SPDX identifiers across file families:

- core protocol files are commonly `CC0-1.0`,
- some verifier files are `MIT`,
- `Batch.sol` is `GPL-3.0-or-later`.

Keep the existing SPDX choice per file family unless the task explicitly requires harmonization.

## Collaboration Rules For Agents

When multiple agents or sequential agents work in this repo, optimize for handoff quality.

### Before changing code

- identify the narrowest set of files that actually matter,
- read both the implementation and its nearest tests,
- inspect for upgradeability or proxy implications,
- ask for clarification before implementing if the specification or intended behavior is unclear,
- check whether the task touches user-facing ABI, deployment scripts, or release artifacts.

### While changing code

- preserve existing naming and event/error style,
- prefer extending existing patterns over introducing new abstractions,
- keep security assumptions explicit in comments only when the code is not already obvious,
- do not refactor unrelated areas "while you are here",
- avoid changing vendored dependencies.

### Before finishing

- run the smallest relevant verification commands first,
- run `forge test` for broader confidence on non-trivial changes,
- mention what you did not verify if you could not run it,
- mention whether ABI artifacts or docs were intentionally left untouched.

### If you encounter unrelated changes

- do not revert them,
- do not "clean up" generated output as a side effect,
- work around them unless they directly block the task,
- if they create a real conflict, surface the conflict clearly.

## Change Patterns

### If the task changes `StateOracle`

Expect to inspect and often update:

- `src/StateOracle.sol`
- `test/StateOracle.t.sol`
- `test/integration/StateOracleWithDAVerifierECDSA.sol`
- possibly `README.md`
- possibly `artifacts/` via generation if ABI changed

### If the task changes role or ownership behavior

Expect to inspect and often update:

- `src/StateOracleAccessControl.sol`
- `test/StateOracleAccessControl.t.sol`
- any deployment or initialization flow that assumes the current role setup

### If the task changes verifier logic

Expect to inspect and often update:

- relevant file under `src/verification/**`
- the corresponding test file
- `README.md` if the operational meaning changed

### If the task changes deployment flow

Expect to inspect and often update:

- relevant `script/*.s.sol`
- `README.md`
- `test/integration/DeployCoreWithStaging.t.sol`
- env var documentation

## Minimal Completion Checklist

Before considering the work done, verify that you have answered these:

- Did I change only the necessary files?
- Did I preserve proxy and storage safety?
- Did I preserve the owner/`DEFAULT_ADMIN_ROLE` invariant?
- Did I add or update the nearest tests?
- Did I run relevant Foundry commands?
- Did I avoid editing generated files by hand?
- Did I update docs or ABI artifacts if the public interface changed?

## Preferred Agent Output

When handing work back to a user or another agent, include:

- what changed,
- why the change belongs in those files,
- what tests were run,
- what was intentionally not changed,
- any remaining risk around upgradeability, permissions, or deployment assumptions.

Concise and explicit beats comprehensive but vague.
